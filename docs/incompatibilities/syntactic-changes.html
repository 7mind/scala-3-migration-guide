Syntactic Changes

Restricted Keywords

Dotty introduces a number of keywords. Some are said regular which means that you cannot use them as identifiers anymore. The others are said soft, meaning that they are not restricted. You can still use the open identifier, for instance.

For the matter of migrating from Scala 2 to Scala 3, the keywords that can cause compilation errors are:

  • enum
  • export
  • given
  • then
  • =>>
  • ?=>

For instance, the following piece of code does not compile anymore:

object given {
  val enum = ???

  println(enum)
}

There is a straightforward solution which is to backquote the keywords when used as identifiers.

Dotty migration rewrite

The Dotty compiler can apply this simple rewrite by itself, with the -source:3.0-migration -rewrite options:

object `given` {
  val `enum` = ???

  println(`enum`)
}

In some cases the compiler is not able to apply the solution by itself but then you can add the -explain option and it will point you towards the solution.

With the following line of code:

def then(): Unit = ()

It prints the following explanation:

-- [E040] Syntax Error: src/main/scala/example.scala:2:6 ------------
2 |  def then(): Unit = ()
  |      ^^^^
  |      an identifier expected, but 'then' found

Explanation
===========

If you want to use 'then' as identifier, you may put it in backticks: `then`.

Procedure Syntax

Procedure syntax has been deprecated in Scala 2 for a while and it is dropped in Scala 3, making the following code illegal:

trait Foo {
  def print()
}

object Bar {
  def print() {
    println("bar")
  }
}

Scala 2.13 deprecation

When compiled in Scala 2 with the -deprecation option it produces the following warnings:

src/main/scala/procedure-syntax.scala:2:14: procedure syntax is deprecated: instead, add `: Unit` to explicitly declare `print`'s return type
  def print()
             ^
src/main/scala/procedure-syntax.scala:6:15: procedure syntax is deprecated: instead, add `: Unit =` to explicitly declare `print`'s return type
  def print() {
              ^

Dotty migration rewrite

When compiled with Dotty and the -source:3.0-migration -rewrite options, the code is rewritten into:

trait Foo {
  def print(): Unit 
}

object Bar {
  def print(): Unit = {
    println("bar")
  }
}

Scalafix rule

Alternatively you can use the Scalafix rule named ProcedureSyntax.

For the case of secondary constructors, which is not handled by the ProcedureSyntax rule, you can use fix.scala213.ConstructorProcedureSyntax of ohze/scala-rewrites/. It turns def this(...) { this(...) } into def this(...) = this(...).

Parentheses Around Lambda Parameter

When followed by its type, the parameter of a lambda is now required to be enclosed in parentheses, making the following Scala 2 code illegal:

val f = { x: Int => x * x }

Dotty migration rewrite

When compiled with Dotty and the -source:3.0-migration -rewrite option it is rewritten into:

val f = { (x: Int) => x * x }

Scalafix rule

The fix.scala213.ParensAroundLambda rule of ohze/scala-rewrites can also fix the incompatibility.

Open Brace Indentation For Passing An Argument

In Scala 2 it is possible to pass an argument after a new line by enclosing it into braces. Although valid, this style of coding is not encouraged by the Scala style guide and it is no longer supported in Scala 3.

This syntax is now invalid:

test("my test")
{
  assert(1 == 1)
}

The preferable solution is to write:

test("my test") {
  assert(1 == 1)
}

Dotty migration rewrite

The Dotty migration rewrite is rather to indent the first line of the block.

test("my test")
  {
  assert(1 == 1)
}

This rule applies to other patterns as well, such as refining a type after a new line.

type Bar = Foo
  {
  def bar(): Int
}

Wrong indentation

The Scala 3 compiler now requires correct indentation.

The following pieces of code that compiled in Scala 2 does not compile anymore, because of the indentation.

def bar: (Int, Int) = {
  val foo = 1.0
  val bar = foo
    (1, 1)
}

val foo =
  Vector(1) ++
Vector(2) ++
  Vector(3)

We suggest using one of the many formatting tools, among which scalafmt, scalariform or the IntelliJ Scala formatter and others.

Beware that these tools need proper configuration, and that some of them are opinionated. They may change the entire code style of your project.

_ As A Type Parameter

This incompatibility is inspired by this commit in the dotty-staging fork of lihaoyi/fastparse project.

The usage of the _ identifier as a type parameter is permitted by the Scala 2, even if it has never been mentioned in the Scala 2 specification. It is used in the fastparse API, in combination with a context bound, to declare an implicit parameter.

def foo[_: Foo]: Unit = ???

Here, the method foo takes a type paramater _ and an implicit parameter of type Foo[_] where _ refers to the type parameter, not the wildcard symbol.

Martin Odersky described this pattern as a "clever exploit of a scalac compiler bug" (source).

The Scala 3 compiler does not permit this pattern anymore:

-- [E040] Syntax Error: src/main/scala/anonymous-type-param.scala:4:10
4 |  def foo[_: Foo]: Unit = ()
  |          ^
  |          an identifier expected, but '_' found

The solution is to give the parameter a valid identifier name, for instance T.

+ And - As Type Parameters

+ and - are not valid identifiers for type parameters in Scala 3, since they are reserved for variance annotation.

You cannot write def foo[+] or def foo[-] anymore.

-- Error: src/main/scala/type-param-identifier.scala:2:10 
2 |  def foo[+]: +
  |          ^
  |          no `+/-` variance annotation allowed here

The solution is to choose another valid identifier, for instance T.

However, + and - still are valid type identifiers in general. You can write type +.
